🔀 Git Client Integration — Implementation Notes

Terax AI Terminal • Sidebar Git Client • May 2025

Architecture Overview

The Git client is built as a native Rust backend (via git2 crate) with a React frontend integrated into the existing right sidebar panel. The right sidebar now has two tabs: Explorer and Git.

  • Backend: src-tauri/src/modules/git.rs — All git operations via libgit2
  • Frontend API: src/modules/git/lib/api.ts — Typed Tauri invoke wrappers
  • State Hook: src/modules/git/lib/useGitStatus.ts — Polling hook (3s interval)
  • Components:

Design Decisions

DECISION

Native git2 over CLI shelling-out

Used the git2 Rust crate (libgit2 bindings) rather than shelling out to git CLI commands. This provides:

  • No dependency on the user having git installed in PATH
  • Structured data without parsing text output
  • Better error handling and cross-platform consistency
  • No shell injection risk (security)

Tradeoff: ~5MB binary size increase from libgit2. No support for git hooks or .gitconfig aliases.

DECISION

Sidebar tab placement (right panel, not left)

The Git panel is tabbed within the right sidebar alongside the File Explorer, not added to the left VerticalTabBar. Reasoning:

  • The left sidebar is a tab bar for workspace views (terminal/editor/preview) — a different conceptual layer
  • The right sidebar is the file-context panel — Git fits naturally here as it's about file state
  • VS Code uses this same pattern (Explorer + Source Control as sidebar tabs)
DECISION

Polling for status (3s interval)

The git status is polled every 3 seconds rather than using file system watchers or IPC events.

  • Simpler implementation, fewer edge cases
  • File watchers on large repos can be noisy and expensive
  • 3s is responsive enough for interactive use without pegging CPU
  • Immediate refresh is triggered after any explicit git action (stage, commit, etc.)
DECISION

Inline SVG icons (no new icon library)

Used inline SVGs for git-specific icons rather than adding another icon package. The existing app uses @hugeicons/react but it doesn't have specialized git icons (branch, merge, etc.). Inline SVGs keep the dependency footprint minimal.

Deviations

DEVIATION

No push/pull/fetch from UI (intentional)

Remote operations (push, pull, fetch) are not exposed in the UI despite the backend being capable. Reasons:

  • These require SSH key or credential management which is complex
  • Terminal is right there — users can git push directly
  • Avoids silent network operations that might fail
  • Can be added later as a phase 2 feature
DEVIATION

No inline diff viewer (yet)

The diff API endpoints exist in the backend (git_diff, git_diff_file) but the frontend currently shows only the file list, not inline diffs. This keeps the initial implementation focused on the core workflow (status → stage → commit) while the diff infrastructure is ready for a future diff viewer component.

Tradeoffs

TRADEOFF

git2 v0.19 vs v0.21

OptionProsCons
git2 0.19 ✓Well-tested, stable API, proven Windows supportOlder libgit2 (1.8.1)
git2 0.21Latest libgit2 featuresMay have breaking API changes, less battle-tested

Chose 0.19 for stability. Cargo will pull it in automatically from the lock file.

TRADEOFF

Sidebar state management: local useState vs global store

OptionProsCons
Local useState ✓Simple, self-contained, no new dependenciesTab state resets on sidebar collapse/expand
Zustand/global storePersistent tab selection, shareable stateOver-engineering for a simple toggle

Chose local state. The tab defaulting to "Explorer" on re-mount is acceptable UX.

TRADEOFF

Diff rendering approach: print() vs foreach()

The git2::Diff::foreach() API takes 4 separate closures which causes Rust borrow-checker conflicts (multiple mutable borrows). Instead used diff.print(DiffFormat::Patch) with a single closure that handles all line types. This is slightly less ergonomic but compiles correctly.

Open Questions

QUESTION

Should the active sidebar tab persist across sessions?

Currently, the sidebar always starts on "Explorer" when the app opens. Should this be saved to the Tauri store and restored? Low priority but easy to add.

QUESTION

Discard all — should there be a confirmation dialog?

Currently "Discard all changes" executes immediately. This is destructive and irreversible. Should we add a confirmation dialog? (Individual file discard also has no confirmation, matching VS Code's behavior of discard-on-click.)

QUESTION

Git log depth — 50 commits enough?

The log view currently loads 50 commits. Should this be paginated/infinite-scroll, or is 50 a reasonable cap for a sidebar panel?

QUESTION

Should "Initialize Repository" be behind a confirmation?

When no repo is detected, we show an "Initialize Repository" button. This creates a .git directory which could be surprising. Currently it's one-click.

Performance Optimizations (Large Repo Support)

DECISION

Adaptive polling interval based on repo size

The useGitStatus hook now adapts its polling interval based on the number of entries returned:

  • <100 entries: 3s (fast, same as before)
  • 100-500 entries: 8s (medium repos)
  • 500+ entries: 15s (large repos — prevents UI freeze from overlapping requests)

Additionally: the gitIsRepo() check is now cached after the first successful confirmation and only re-checked if the path changes or git_status returns a "Not a git repository" error. This removes one redundant IPC call per poll tick.

An inFlightRef guard prevents overlapping requests — if a previous poll hasn't returned yet, the next tick is skipped entirely.

DECISION

Conditional recurse_untracked_dirs in Rust backend

The git_status command previously used recurse_untracked_dirs(true) unconditionally. On large repos with many untracked directories (e.g., build artifacts, node_modules not yet gitignored), this causes libgit2 to walk the entire file tree — potentially millions of files.

New behavior: A fast first pass with recurse_untracked_dirs(false) is performed. If the result has <300 entries, a second pass with full recursion is done for accurate listing. Large repos get the fast result (directories shown as single entries rather than expanded).

Tradeoff: On large repos, untracked directories appear as collapsed folder entries (e.g., build/) rather than listing every file inside. This matches VS Code's behavior for performance.

DECISION

Progressive rendering for file lists

In list mode, only the first 100 files are rendered initially. A "Show more" button renders the next 200 at a time. This prevents DOM thrashing when a repo has thousands of changed files.

In tree mode, folders start collapsed by default when the entry count exceeds 200. This avoids rendering thousands of nested DOM nodes on mount.
